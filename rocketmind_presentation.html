GREL — Graph-based Reasoning & Evolutionary Learning

The system that gets smarter with every query

What is GREL?

GREL is the learning layer that sits on top of the retrieval system. It continuously evaluates which strategy works best for each type of query and evolves over time.

5 Competing Strategies

StrategyMethodCost
A Decision TreeRule-based pattern matchingFree
B TF-IDF RAGKeyword retrievalLow
C Tool-UseLLM native tool selectionMedium
D Full ReasoningMulti-step ReAct loopHigh
E Hybrid RetrievalGraph + Vector + LexicalMedium

Tournament Evolution

Week 1-2: TOURNAMENT
// Run ALL strategies in parallel
// Log which one wins for each query pattern

Week 3-4: SHADOW
// Run winner + 1-2 shadows for validation
// Verify winner consistency

Month 2+: CONVERGED
// Run winner only (shadow 10% for drift)
// Cost drops 60-80% while quality holds

Learning Signals

Positive

Correct tool selected. User accepted answer. Low latency.

Negative

Wrong tool. User escalated. Timeout. Low confidence.

RIPER Engine — 5-Phase Structured Reasoning

For COMPLEX queries that need multi-step analysis

When RIPER Activates

// RequestClassifier output:
QUICK → Direct ReAct (1 loop, <500ms)
STANDARD → RIPER Lite (R+E+R, skip I+P)
COMPLEX → Full RIPER (all 5 phases)

5 Phases

PhaseActionOutput
R RESEARCHGather all KB context, entity data, graph neighborsResearch artifact with evidence
I INNOVATEGenerate multiple answer strategies from research2-3 candidate approaches
P PLANSelect best strategy, plan tool execution orderExecution plan with dependencies
E EXECUTERun tools in dependency order (parallel where possible)Tool results + confidence score
R REVIEWQuality check: hallucination, completeness, accuracyFinal response + quality score

GREL — Gather, Reason, Evaluate, Learn

The evolutionary learning system with 2-wave parallel execution

4 Phases

GATHER: Run strategies in 2 waves
  Wave 1 (cheap, parallel):
    Decision Tree + TF-IDF + Hybrid Retrieval
    Timeout: 15s per task
  
  if Wave 1 confidence ≥ 0.75 → skip Wave 2
  
  Wave 2 (expensive, only if needed):
    Claude Tool-Use + Full Reasoning

REASON: Merge results from best strategy
  Weighted by: confidence × latency_penalty

EVALUATE: Score final answer quality
  Criteria: completeness, accuracy, relevance

LEARN: (async, after response sent)
  Record: which strategy won, for what query type
  Update: tournament weights + Kafka event

Wave Executor Architecture

WaveExecutor:
  Wave 1: [Task A, Task B, Task C] → asyncio.gather()
  ↓ results + consensus check
  Wave 2: [Task D, Task E] → asyncio.gather()
  ↓ skip if Wave 1 sufficient
  Progress: SSE callback per task status change

3-Tier Escalation System

Tier 1: Brain (KB + Graph + Tools)
  Score = evidence × 0.4 + entity_resolved × 0.3
        + live_data × 0.2 + classifier_conf × 0.1
  score ≥ 0.70 → RESPOND
  score < 0.70 → ESCALATE to Tier 2

Tier 2: Code Intelligence
  Deep code analysis + retry with enriched context
  score ≥ 0.60 → RESPOND
  score < 0.60 → ESCALATE to Tier 3

Tier 3: Safe DB Query
  Template-driven SQL against prod slave (read-only)
  score ≥ 0.50 → RESPOND
  score < 0.50 → ESCALATE to Human

RALPH Self-Correction

// Runs AFTER ReAct produces response
REVIEW: Check response against KB evidence
ANALYZE: Identify gaps, contradictions, claims
LEARN: Record what worked / what failed
PRIORITIZE: Rank corrections by severity
HARMONIZE: Rewrite if quality < threshold

Max 3 cycles. Cycle 3 = escalate to human.

Tools → Agents → Skills → Actions — Full Chain

How a query becomes a tool call becomes an action becomes a response

The 4-Layer Stack

TOOL = an API wrapper with a schema
  wallet_balance(seller_id: int) → {balance: float}
  cancel_order(order_id: int) → {status: str}

AGENT = a domain specialist with a tool subset
  billing_wallet agent → [billing_query, wallet_balance,
                       initiate_refund, cancel_order]
  KB scope: ["billing", "wallet"] → 2K chunks only

SKILL = a named workflow bound to an agent
  cancel_and_refund.skill:
    triggers: ["cancel order", "cancel karo", "refund"]
    agent: billing_wallet
    actions: [step1, step2, step3]

ACTION = one step within a skill (7 types)
  api_call → execute a tool
  prompt   → ask LLM to reason
  handoff  → pass context to another agent
  notification → send webhook/event
  workflow → trigger external workflow (N8N)
  respond  → format and return answer
  internal → local logic (check, compute, validate)
