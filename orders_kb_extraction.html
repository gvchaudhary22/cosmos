Pillar 1: Schema Extraction

How orders table YAML is created from the PHP model

Source: PHP Model

// app/Repositories/Order.php
class Order extends ArchiveBaseModel
{
    use OrdersWeight, SoftDeletes;

    protected $table = 'orders';
    protected $connection = 'mysql';

    protected $fillable = [
        'company_id', 'channel_id',
        'channel_order_id', 'order_date',
        'customer_name', 'customer_address',
        'customer_pincode', 'customer_phone',
        'payment_method', 'sub_total',
        'weight', 'dimensions', 'status',
        'rto_prediction', ...
    ];

    // Status constants
    const NEW_ORDER = 1;
    const INVOICED = 2;
    const READY_TO_SHIP = 3;
    const SHIPPED = 6;
    const DELIVERED = 7;
    const CANCELLED = 5;
}

Extraction Process

1
Scanner reads Order.php → extracts $table, $fillable, $connection, constants
2
Reads database migration files → extracts column types, indexes, foreign keys
3
Reads const values → maps status integers to names (1=NEW, 6=SHIPPED, etc.)
4
Reads route files → finds which APIs read/write this table
5
Merges into high.yaml (3,729 lines) then chunks into high/ folder (12 files)

How it helps COSMOS

When a user asks "what are the valid order statuses?", the retriever finds states_constants.yaml which lists all 105 status values. Without this, the LLM would guess common statuses and miss Shiprocket-specific ones like UNMAPPED(10) or UNFULFILLABLE(11).

How 660 orders APIs are extracted from route files → tool assignment

How tool assignment works

1
Route scanning: pillar3_bulk_generator_v1 reads all route files, extracts method+path+controller
2
Domain classification: Path /api/v1/orders/* → domain = orders
3
Tool assignment: Method+subdomain: POST+create → orders_create. GET+list → orders_list
4
Agent assignment: All orders tools → order_ops_agent
5
Request schema: Scanner reads AdHocRequest.php validation rules → fills required/optional params
6
Examples: Creates param_extraction_pairs: "create order for customer" → {endpoint, params}

How it helps COSMOS

When a user asks "which API creates orders?", COSMOS retrieves mcapi.v1.orders.create.post:overview which has the exact endpoint, controller, and parameters. How each file is created

purpose: "Create new order with address validation, product items, COD/Prepaid"
required_inputs:
  - order_id (string): unique channel_order_id
  - billing_pincode (string): required, digits:6  ← from AdHocRequest.php validation
sync_async: sync

3

  hinglish: ["naya order create karo", "ek order banana hai"]

4

  1. validate inputs (AdHocRequest validation rules)
  2. check address serviceability (pincode → courier)
  3. calculate rates (freight engine)
  4. store to orders + order_products tables
  5. trigger SaveRTOprediction (async job)
5
eval_cases.yaml — Test cases for retrieval quality
positive: "Create a new order" → action.orders.create_order
regression: "Ship this order" must NOT route to create_order
unsafe: "Create 10000 orders" → reject (bulk without validation)

Pillar 7: Workflow Runbook Creation

How the order creation pipeline becomes a 13-file runbook

Why workflows exist (that P6 actions don't cover)

P6 tells youP7 tells you
"create_order" is one actionOrder creation is a 7-state pipeline with decision branches
What inputs are neededWhat happens AFTER: validated → address_verified → rate_calculated → ready_to_ship
What can failException paths: what if address service is down? what if duplicate order?
How to rollbackOperator playbook: diagnose → verify → act → escalate

Key workflow files and what they contain

1
state_machine.yaml
7 states: order_placed → validated → address_verified → rate_calculated → ready_to_ship
13 allowed transitions + 3 invalid transitions (cannot go backwards)
2
decision_matrix.yaml
decisions:
  - COD vs Prepaid: COD needs serviceability check, Prepaid has no limit
  - Domestic vs International: international needs HSN codes + customs
  - Channel sync vs Manual: different validation rules per channel
3
operator_playbook.yaml — For ICRM support operators
diagnose:
  1. Check order status in DB
  2. Verify address serviceability
  3. Check if channel sync completed
act:
  - "Order stuck in validation_failed" → check which field failed, ask seller to correct
  - "Address not serviceable" → suggest alternate pincode or different courier
4
user_language.yaml — How sellers/ICRM actually talk about this
seller_wording: ["order create nahi ho raha", "order stuck hai"]
hinglish: ["address galat hai", "pincode serviceable nahi hai"]
confusing_synonyms:
  - "create" vs "import" (different flows)
  - "cancel order" vs "cancel shipment" (very different!)

How it helps COSMOS

When a user asks "why is my order stuck?", COSMOS retrieves the operator_playbook which gives step-by-step diagnosis. It also retrieves the state_machine to show which state the order is in and what transitions are valid next. Without this, the LLM would give generic advice instead of Shiprocket-specific troubleshooting steps.

How KB Generates Tools, Agents & Skills

The KB-Driven Registry reads graph nodes and creates runtime objects

Step 1: Graph nodes exist (from training pipeline)

graph_nodes table:
  | node_id              | node_type        | domain  | label           |
  |----------------------|------------------|---------|-----------------|
  | tool:orders_create   | tool             | orders  | orders_create   |
  | tool:orders_list     | tool             | orders  | orders_list     |
  | agent:order_ops_agent| agent            | orders  | order_ops_agent |
  | intent:orders_create | intent           | orders  | orders_create   |
  | action:create_order  | action_contract  | orders  | create_order    |

graph_edges table:
  | source_id                     | target_id            | edge_type        | weight |
  |-------------------------------|----------------------|------------------|--------|
  | api:mcapi.v1.orders.create    | tool:orders_create   | implements_tool  | 1.5    |
  | api:mcapi.v1.orders.create    | agent:order_ops_agent| assigned_to_agent| 1.2    |
  | action:create_order           | api:orders.create    | calls_api        | 1.6    |

Step 2: KB-Driven Registry reads graph → creates runtime objects

// kb_driven_registry.py: _sync_from_graph()

// Query 1: Find all tools
SELECT * FROM graph_nodes WHERE node_type = 'tool' AND domain = 'orders'
→ Creates 5 DynamicTool objects

// Query 2: Find the agent
SELECT * FROM graph_nodes WHERE node_type = 'agent' AND domain = 'orders'
→ Creates DynamicAgent("order_ops_agent")

// Query 3: Find intents (= skills)
SELECT * FROM graph_nodes WHERE node_type = 'intent' AND domain = 'orders'
→ Creates DynamicSkill objects

// Query 4: Link tools to agent via shared API edges
SELECT tool_id, agent_id FROM graph_edges
WHERE edge_type IN ('implements_tool', 'assigned_to_agent')
→ Links: orders_create tool ↔ order_ops_agent (349 shared APIs)

Step 3: At runtime, when user says "create order"

1. Intent classification: "orders_create" (from intent_map Hinglish phrases)
2. Agent selection: order_ops_agent (matches domain "orders")
3. Tool selection: orders_create (matches intent "orders_create")
4. Action contract: action.orders.create_order (has preconditions, side effects)
5. API selection: POST /api/v1/orders/create (primary endpoint)
6. Execute with: FACTUALITY_PROMPT + source-attributed context
7. Verify with: RALPH grounding check + HallucinationGuard

The key insight

The KB doesn't just store information — it generates the tools and agents that COSMOS uses at runtime. When you add a new API to the KB, it automatically becomes a tool under the right agent. When you add an action contract, it automatically gets precondition checking. This is why the KB is the brain, not just documentation.
