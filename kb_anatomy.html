2. How One API File Defines Everything

Every field in high.yaml maps to a runtime concept:

# apis/mcapi.v1.orders.create.post/high.yaml

overview:
  api:
    method: POST
    path: /api/v1/orders/create
    controller: CustomController@storeAdHoc
  classification:
    domain: orders ← routes to order_ops_agent
    intent_primary: orders_create ← activates "cancellation" skill

tool_agent_tags: ← THIS SECTION DEFINES TOOL + AGENT + SKILL
  tool_assignment:
    tool_candidate: orders_create ← TOOL NAME
    risk_level: high ← ACTION RISK (needs approval)
    read_write_type: WRITE ← ACTION TYPE
  agent_assignment:
    owner: order_ops_agent ← AGENT OWNER
    secondary: [support_agent] ← BACKUP AGENT
  intent_tags:
    primary: orders_create ← SKILL/INTENT NAME
    secondary: [orders_lookup, orders_workflow]

request_schema: ← TOOL INPUT PARAMS
  contract:
    required:
      - name: order_date     type: date     validation: required|date
      - name: channel_id    type: string   validation: required|exists:channels
      - name: billing_phone type: numeric  validation: required|digits:10
      # ... 28 total required params

response_fields: ← TOOL OUTPUT
  fields: [id, sku, name, quantity, color, size, channel_code, image]
  events_triggered: [AdminReportRequested] ← SIDE EFFECTS
  jobs_dispatched: [AutoAssignAWBChannels]

examples: ← SKILL TRIGGERS
  param_extraction_pairs:
    - query: "run orders create for this account"
      params: {endpoint: /api/v1/orders/create, method: POST}

3. KB Section → Runtime Concept Mapping

KB Section YAML Field Runtime Concept Example Value How It's Used
tool_assignment tool_candidate TOOL orders_create Graph node tool:orders_create. Should auto-register in ToolRegistry.
tool_assignment risk_level ACTION RISK high Determines if supervisor approval is needed before execution.
tool_assignment read_write_type ACTION TYPE WRITE WRITE = action (needs approval). READ = safe to auto-execute.
agent_assignment owner AGENT order_ops_agent Graph node agent:order_ops_agent. Owns this tool + all order APIs.
intent_tags primary SKILL orders_create Graph node intent:orders_create. Triggers order creation workflow.
request_schema contract.required TOOL PARAMS 28 fields (order_date, channel_id...) What the tool needs as input. Validates user-provided data.
response_fields fields TOOL OUTPUT [id, sku, name, quantity...] What the tool returns. Used to format response to user.
response_fields events_triggered SIDE EFFECTS [AdminReportRequested] Events fired after tool execution. Shown in response context.
examples param_extraction_pairs SKILL TRIGGERS "run orders create..." Example queries that activate this skill. Used for intent matching.
overview.classification domain AGENT SCOPE orders Domain routing: query about "orders" → order_ops_agent.

6. The Gap: KB Has It All, Code Doesn't Read It

What KB Provides (source of truth)

  • ✓ tool_candidate: "orders_create" with params, risk, R/W type
  • ✓ agent_assignment: "order_ops_agent" with tool scope
  • ✓ intent_tags: "orders_create" with example triggers
  • ✓ request_schema: 28 required fields with validation rules
  • ✓ response_fields: output fields + events + jobs
  • ✓ risk_level: determines approval flow

What Code Does Instead (hardcoded)

  • ✗ 23 tools defined as Python classes (read_tools.py, write_tools.py)
  • ✗ 18 agents defined as Python dicts (agent_registry.py)
  • ✗ 10 skills defined as Python dicts (skill_registry.py)
  • ✗ Tool params hardcoded, not from KB request_schema
  • ✗ Agent scope hardcoded, not from KB agent_assignment
  • ✗ Approval rules hardcoded, not from KB risk_level

The Fix: sync_from_graph()

Add new API to KB → graph gets tool node → tool auto-registers → no code change needed
