What is this platform?

A single system that connects all Shiprocket services and lets AI (plus rules) read data, take actions, and automate work — so employees spend less time searching and more time deciding.

Anyone can create a specialised AI agent using plain language and ready-made building blocks — no coding required.

The Goal — in plain words

📚

Capture what we know

Every team at Shiprocket has years of experience and unwritten rules. Right now, that knowledge lives in people's heads, scattered docs, and Slack messages. We put it all in one place.

⚡

Make it instantly usable

Instead of asking a senior colleague or digging through dashboards, any employee can just ask a question in plain language and get the right answer in seconds.

🔄

Keep getting smarter

Every time someone uses it, it learns. The more people use it, the better it gets — automatically — without anyone having to manually update it.

Knowledge Architecture — 8 Pillars

Every agent answer is grounded in one or more pillars. No hallucination — only retrieved facts. Each pillar answers a different question an operator might ask.

P1

Schema

What data exists?
676 tables, 105 status constants, column types, validation rules, state transitions.

Why agents need it: Before touching any data, an agent must know which table holds it, what values are valid, and which status codes mean what.

Query: "What is order status 6?"
→ P1 returns: SHIPPED, transition from status 3 only
P2

Workflows

How does a process work end-to-end?
State machines, decision trees, operator playbooks, exception paths, escalation rules.

Why agents need it: P1 tells you the fields. P2 tells you the correct sequence of steps. Without P2, an agent can look up data but can't guide a multi-step operation correctly.

Query: "How to process a prepaid return?"
→ P2 returns: step 1→2→3, conditions, refund trigger point
P3

APIs → Tools

What API can I call?
5,617 endpoints with method, path, controller, params, tool assignment and agent mapping.

Why agents need it: Every agent action that reads or writes data must go through an API. P3 is the bridge between intent and invocation.

Query: "Cancel this order"
→ P3 maps to: POST /icrm/order/cancel
P4

Pages + Roles

Where is this field on the UI?
24 pages, field→API→table.column traces, role-based visibility, UI component mapping.

Why agents need it: Operators describe problems using UI terms ("the weight field on the NDR page"). P4 translates UI language into backend reality.

Query: "Weight field on NDR page"
→ P4 traces to: shipments.charged_weight
P5

Module Docs

What code handles this?
713 files: controllers, services, jobs, routes, business rules per module.

Why agents need it: For debugging and dev tasks, the agent must know which service owns the logic, which job processes async work, and where business rules live.

Query: "Who handles NDR auto-close?"
→ P5 returns: NdrService::autoClose()
P6

Action Contracts

What should the agent do — and how safely?
25 actions × 11 files: preconditions, execution graph, side effects, rollback, approval mode.

Why agents need it: Defines the exact contract for every write operation — what must be true before acting, what changes after, and whether human approval is required.

Query: "Initiate RTO for AWB 12345"
→ P6: check preconditions → confirm → execute → notify
P7

Runbooks

Why did this happen + how to fix it?
Operator playbooks, incident runbooks, decision matrices, exception paths for 8 domains.

Why agents need it: Captures tribal knowledge that isn't in code — the implicit rules ops teams follow when things go wrong. Converts senior expertise into retrievable context.

Query: "Delivery failed 3 times, what now?"
→ P7: NDR runbook → check reason → reattempt or RTO
P8

Negative Routing

Don't confuse X with Y.
100 negative examples: ambiguous Hinglish queries, similar-sounding operations, common misroutes.

Why agents need it: "Cancel order" ≠ "cancel shipment". P8 prevents the most dangerous class of errors — confidently doing the wrong thing.

Query: "cancel karo" (ambiguous)
→ P8: disambiguate → ask "order or shipment?"

Stage 0 — MCP Chat → Mars (Entry Point)

The operator types a query in Lime Frontend (MCP Chat at /chat/[id]). The message is sent via SSE streaming to Mars at POST /api/v1/chat/stream.

Mars validates the JWT token, resolves the active project/repo context, injects skill prompts from the Skillloader (19 built-in skills), then forwards the enriched request to RocketMind (Cosmos) at POST /cosmos/api/v1/hybrid/chat.

// Lime → Mars
POST /api/v1/chat/stream
{ message, conversation_id, mode }

// Mars enriches + forwards to Cosmos
POST /cosmos/api/v1/hybrid/chat
{ query, repo_id, company_id,
  skills: ["safety-context","module-context"…],
  user_id, session_id }

Stage 1 — Request Classifier (3-Dimension Classification)

Before any retrieval, every query is classified along 3 axes using lightweight keyword matching (no LLM call — fast):

① DOMAIN

Which part of the system is this about?
ORDERS · SHIPMENTS · NDR · BILLING · SELLER · RETURNS · SYSTEM · GENERAL

"cancel order 123" → ORDERS

② COMPLEXITY

How deep should retrieval go?
QUICK — skip Wave 2
STANDARD — conditional deep
COMPLEX — fire all pipelines

"why did NDR fail?" → COMPLEX

③ MODE

What kind of operation?
LOOKUP — read-only info
ACTION — state-changing
DIAGNOSTIC — trace / debug

"initiate refund" → ACTION

💡 QUICK queries skip Wave 2 entirely — direct to LLM with Wave 1 context. Saves ~400ms per simple lookup.

Stage 2 — Wave 1: Parallel Probe (5 Pipelines Simultaneously)

All 5 pipelines fire in parallel using asyncio.gather(). Each probe takes ~50–200ms. The results are merged into a unified context object.

🔢 Intent Classifier

Identifies what the user wants — lookup, cancel, track, refund, explain, debug. Maps to structured intent tags used to score retrieved results.

"cancel order" → intent: cancel_order
entity: order

🧲 Vector Search

Semantic similarity search against 31K+ embedded KB chunks using pgvector. Uses Scoped Retrieval — each agent only searches its domain's ~2–5K chunks, not all 31K.

NDR agent: 31K → ~2K chunks
Top-K: 8 chunks returned

🕸️ GraphRAG Probe

Lightweight graph lookup — finds the entity node and its 1-hop neighbours in the 867-node knowledge graph. Surfaces relationships vector search misses.

"order" node → linked to shipment, NDR, billing nodes

🗂️ Entity Probe

Directly fetches the Entity Hub doc for the matched entity — a pre-merged 500–800 token summary combining P1+P3+P6+P7. Always fast, always in-context.

entity: order → hub/orders.yaml loaded in full

🚫 Negative Routing (P8)

Checks if the query matches a known negative example. If "cancel order" vs "cancel shipment" ambiguity is detected, the probe flags it before any action is attempted.

"cancel karo" → AMBIGUOUS → ask to clarify

Stage 3 — Wave 2: Conditional Deep Retrieval (Only for STANDARD + COMPLEX)

If Wave 1 probes return low confidence, or the classifier said COMPLEX, Wave 2 fires deeper pipelines:

🕸️ GraphRAG Deep

Multi-hop graph traversal — follows relationships 2–3 hops deep. Finds non-obvious connections like order → shipment → NDR → courier → delay reason.

traversal: order→shipment→NDR→delay_reason

🧠 LangGraph (Wave 3)

Stateful reasoning — a mini state-machine that iterates retrieval. If first attempt low confidence, it reformulates the query and retries up to 3 times with adjusted strategy.

confidence < 0.6 → reformulate → retry

🔷 Neo4j (Wave 4)

Targeted graph database traversal for relationship-heavy queries. Used when GraphRAG in-memory graph is insufficient — queries full Neo4j up to configurable depth.

max_depth: 3 · timeout: configurable

Stage 4 — RIPER Engine: Structured LLM Assembly

All Wave 1 + Wave 2 context is merged and passed to the LLM. For COMPLEX queries, RIPER structures the reasoning in 5 phases. For QUICK/STANDARD, RIPER Lite (4 phases) is used.

R — Research

Summarise what was retrieved from all pillars. What facts are available? What is still unknown?

I — Innovate

Consider multiple approaches. Which pillar combination best answers this? Are there alternative interpretations? (COMPLEX only)

P — Plan

Structure the response. Decide: cite which sources, state preconditions if ACTION, include which workflow steps.

E — Execute

Generate the final response. Factuality rules enforced: only facts from KB context, cite with [1][2][3] markers, never invent IDs.

R — Review

Self-check before sending: did I answer all sub-intents? Are citations correct? Is the action safe to proceed?

Stage 5 — RALPH Engine: Post-Response Self-Correction Loop

RALPH runs after RIPER generates a response. It acts as an independent quality verifier — can trigger a retry or escalate to human if needed.

🪞 Reflect

Did the response actually answer what was asked? Checks for: missing intent, hallucination, wrong entity, incomplete answer.

⚡ Act

If verdict = RETRY, identify what went wrong (wrong entity, low confidence, hallucination) and adjust retrieval strategy.

📚 Learn

Records every correction to Kafka → continuous training data. Failed responses become future KB improvements automatically.

🗺️ Plan + Help

If fixable → retry with adjusted approach (max 3 cycles). If still failing → verdict = ESCALATE → route to human operator.

PASS

Response is good — stream to user

RETRY

Fixable — re-run with adjusted strategy (up to 3×)

ESCALATE

Not fixable by AI — route to human operator

Engineering Deep-Dive · Phase 1 of 5

Phase 1 — Request Classification

Every query is 3-dimensionally tagged before a single vector search fires

Class: RequestClassifier

Located in cosmos/app/services/request_classifier.py

Entry point: classify(query: str) → RequestClassification

Returns a 3-axis tag used to route the entire downstream pipeline — controls which waves run, whether RIPER goes full or lite, and agent selection strategy.

Confidence Formula

confidence = min(0.95, 0.5 + total_hits * 0.08)

# Each regex pattern match = +0.08
# Floor: 0.5 (always at least 50% certain)
# Ceiling: 0.95 (never claims perfect certainty)

Three Classification Axes

Axis 1 — Domain (8 values)

ORDERSSHIPMENTSNDR BILLINGSELLERRETURNS SYSTEMGENERAL

Pattern matching. Each domain has regex patterns. Max-scoring domain wins.

Axis 2 — Complexity (3 values)

QUICK

Skip deep waves. ~50ms total. Triggered by: "show/get/find/check", "where is", "status"

STANDARD

Default. Run Wave 1 + conditional Wave 2.

COMPLEX

All deep pipelines. Triggered by: "why", "trace", "how does", multi-domain, multi-part

Axis 3 — Mode (3 values)

LOOKUP

Read-only. No write tools loaded.

ACTION

Write intent. Action contracts loaded (P6). Triggers: cancel/refund/escalate/reassign

DIAGNOSTIC

Root-cause. RIPER full mode forced. Triggers: why/trace/debug/root cause/kyu

Complexity Decision Logic (actual code logic)

if complex_hits >= 2 OR multi_domain OR multi_part:
    complexity = COMPLEX               # fire all deep pipelines

elif quick_hits > 0 AND complex_hits == 0 AND mode != DIAGNOSTIC:
    complexity = QUICK                 # skip Wave 2, go straight to LLM

else:
    complexity = STANDARD              # Wave 1 + conditional Wave 2

# Mode resolution
if action_hits > diagnostic_hits:  mode = ACTION
elif diagnostic_hits > 0:          mode = DIAGNOSTIC
else:                              mode = LOOKUP

Phase 2 — Parallel Wave Execution + Scoped Retrieval

5 parallel probes fire in Wave 1. Deep pipelines activate conditionally in Wave 2. Each probe searches only its agent's domain slice.

Class: WaveExecutor

cosmos/app/services/wave_executor.py

Key invariant: Wave N+1 starts only after Wave N completes. Within a wave, all tasks run via asyncio.gather().

Progress emitted per-task via _emit_progress(wave_id, task_id, status, data) — SSE stream shows wave progress in real-time on Lime UI.

Task State Machine

PENDING → RUNNING → SUCCESS or ERROR or SKIPPED

Consensus checking: all tasks in a wave must succeed for Wave N+1 to start. Conditional skip: skip_wave_if(wave_id, context) → bool — QUICK complexity skips Wave 2 entirely.

Phase 3 — RIPER: Structured LLM Reasoning

When integrated: 5-phase structured reasoning loop transforms retrieved context into a grounded, cited, verifiable response

Class: RIPEREngine

cosmos/app/engine/riper.py

Two execution paths chosen by complexity:

process_full()

R → I → P → E → R
Used for COMPLEX / DIAGNOSTIC

process_lite()

R → P → E → R (skip Innovate)
Used for STANDARD / QUICK

Final response delivered via stream_final_response() — token-level streaming to SSE. Latency estimate: len(tools) × 80ms + 200ms synthesis.

Factuality Enforcement (hardcoded into context)

FACTUALITY_PROMPT — 10 hard rules injected into every RIPER execution:

1. Never fabricate values — only use retrieved context
2. Always cite source (P1–P8 pillar, tool name, DB table)
3. Validate enum values against P1 schema constants
4. Trace all field references to actual column names
5. For actions, confirm approval_mode before executing
6–10. Entity verification, multi-part coverage, Hinglish handling…

The 5 RIPER Phases — What happens in each

R

Research — ResearchArtifact

Extracts: known_facts[], unknowns[], constraints[], entities_found[] from Wave context. Anchors all downstream phases to verified facts only.

Engineering insight: If unknowns is non-empty, RIPER flags for Tier 2/3 fallback before proceeding.

I

Innovate — InnovateArtifact (full mode only)

Generates 3+ approaches with trade-offs. Fields: approaches[], recommended, reasoning. This is what separates a thoughtful answer from a cached lookup.

Skipped in process_lite() for STANDARD/QUICK queries to save ~200ms LLM call.

P

Plan — PlanArtifact

Creates: steps[], success_criteria[], estimated_latency_ms. For ACTION mode, each step maps to a specific tool call with approval gating.

Latency estimate: len(tools) × 80 + 200 ms. Used for timeout budgeting and progress bar on Lime UI.

E

Execute — ExecuteArtifact

Runs the plan via ReActEngine (tool calls). Populates: step_results[], response, confidence (0–1), tools_used[].

Tool calls go through Mars circuit-breaker. If circuit is open, falls back to context-only response with reduced confidence score.

R

Review — ReviewArtifact

Verifies response against success criteria. Fields: criteria_met[], all_intents_addressed, quality_score (0–1), issues[].

If quality_score < 0.7 or all_intents_addressed == False, flags response before passing to RALPH. Not a retry — just a signal for the next phase.

Phase 4 — RALPH: Post-Response Self-Correction

⚠️

When integrated: every response passes through a 5-phase quality loop. RALPH can retry up to 3 times or escalate to human.

cosmos/app/engine/ralph.py

Entry point: evaluate(response, query, context, intents). Returns a verdict + correction type + optionally an adjusted_query for retry.

Correction Types (Enum)

NONE Response passes all checks. Ship it.
MISSING_INTENT Did not address all sub-intents detected by classifier
LOW_CONFIDENCE ExecuteArtifact.confidence < 0.5
HALLUCINATION Context grounding < 30%. Response terms not in KB.
WRONG_ENTITY Answered about a different entity ID than queried
INCOMPLETE Multi-part query: each question needs ≥50 chars answer
ESCALATE AI cannot fix. Routes to human or senior agent.

The 5 RALPH Phases

Reflect

Check 4 signals: context grounding %, intent coverage, confidence score, entity ID match. Produces a CorrectionType verdict.

Act

Formulate fix hypothesis: generate adjusted_query + supplemental context for retry. Maps correction type → repair strategy.

Learn

Call _emit_learning() → LearningInsightEvent → Kafka. Each correction is distillation input for the next model version.

Plan

Execute retry if cycles < MAX_CYCLES. Re-runs Wave search + RIPER with adjusted_query and refined context.

Help

After 3 failed cycles: verdict = ESCALATE. Routed to human supervisor or senior-tier agent.

PASS Verdict

Response ships only after: confidence ≥ 0.5, grounding ≥ 30%, all intents covered, correct entity referenced.

How Grounding Is Calculated

grounding_score = (
    response_terms ∩ knowledge_base_terms
) / total_response_terms

if grounding_score < 0.30:
    correction_type = HALLUCINATION
    verdict = RETRY  # or ESCALATE after cycle 3

This is the core anti-hallucination gate. A response that makes 10 factual claims must have at least 3 of them directly traceable to retrieved KB chunks.

Combined with RIPER's FACTUALITY_PROMPT (which prevents fabrication at generation time), RALPH provides a second independent check at verification time.

Engineering Deep-Dive · Phase 5 of 5

Phase 5 — Learning Loop + Tier Fallback Architecture

Every response either ships or teaches. The 4-tier fallback ensures no query goes unanswered. Every correction feeds the next model version.

4-Tier Fallback — How QueryOrchestrator.execute() escalates

Each tier has a multi-signal gate before advancing. Gate checks: confidence + evidence count + freshness + entity resolution.

Tier 0
~5ms

Semantic Cache

Vector similarity check on recent Q&A pairs. Exact semantic match → skip all waves, return cached response instantly.

Tier 1
~50–800ms

Brain — Wave Execution + RIPER + RALPH

Full pipeline: classify → wave probe → deep retrieval → RIPER assembly → RALPH verification. Handles 85%+ of queries.

Tier 2
+200–600ms

Code Intelligence + LLM Rewrite (1 retry)

If Tier 1 gate fails: codebase_intel searches source code directly. LLM rewrites query for a second attempt. Catches edge cases not in KB.

Tier 3
+variable

Safe DB Tool (template-driven via Mars)

If Tier 2 gate also fails: parameterized SQL templates via safe_db_tool → MARS circuit breaker → real DB. Zero raw SQL generation.

The Learning Loop

1. RALPH correction recorded → LearningInsightEvent
2. _emit_learning() → Kafka topic learning.corrections
3. Mars Learning module: createLearningRecord() persists to DB
4. Pattern detection: cluster similar corrections
5. Improvement proposals surfaced to Manager Panel (Lime UI)
6. Approved proposals → fine-tuning / KB update pipeline

What is the KB?

The Knowledge Base is not documentation. It is a structured data pipeline that converts raw code into executable AI capabilities:

PHP Code → Scanner → YAML Files → Renderer → Embeddings → Graph → Tools + Agents → Claude Response

When an ICRM operator asks a question, the answer comes from retrieved KB data, not from the LLM guessing. The LLM only synthesizes — the KB provides the facts.

Architecture

How Agent, Tool, Skill & Action Relate

A hierarchy: Agent uses Skills, Skills execute Actions, Actions call Tools

Agent (who) — e.g., order_ops_agent
  |
  +-- has: tools_allowed      [order_lookup, cancel_order, update_address, ...]
  +-- has: knowledge_scope    ["orders", "core_orders"] ← filters KB retrieval
  +-- has: instructions       "Verify order exists before any action..."
  +-- has: can_handoff_to     [billing_wallet, shipment_ops, ndr_resolver]
  +-- can use: Skills         [order_lookup, cancellation, ...]
       |
       |
Skill (how) — e.g., order_lookup
  |
  +-- has: triggers          ["order status", "find order", "order kahan hai"]
  +-- has: actions[]         (ordered steps to execute)
       |
       |
Action (one step) — e.g., fetch_order
  |
  +-- type: api_call   → calls a Tool
  +-- type: prompt     → calls LLM
  +-- type: handoff    → transfers to another Agent
  +-- type: respond    → formats answer for user
  +-- type: internal   → logs/updates state
       |
       |
Tool (the actual API call) — e.g., order_lookup
  |
  +-- has: parameters        [{name: "order_id", type: "str", required: true}]
  +-- has: data_source       MCAPI (Shiprocket API via SSO token)
  +-- has: risk_level        LOW / MEDIUM / HIGH
  +-- has: approval_mode     auto / confirm / manual

Think of it as:

Agent = WHO   |   Skill = HOW   |   Action = ONE STEP   |   Tool = THE API CALL

class OrderLookupTool(BaseTool):
    definition = ToolDefinition(
        name="order_lookup",
        category=ToolCategory.READ,
        description="Look up order by ID,
          AWB, or channel order ID",
        parameters=[
            ToolParam(
                "order_id", "str",
                True,
                "Order ID, AWB, or
                 channel order ID"
            ),
        ],
        data_source=DataSource.MCAPI,
        risk_level=RiskLevel.LOW,
    )

    async def execute(self, params, ctx):
        resp = await self.mcapi.get_order(
            params["order_id"],
            headers=ctx.get("headers"),
        )
        return ToolResult(
            success=resp.success,
            data=resp.data,
        )
